# Section Dependency Audit Prompt

Use this prompt for the section-scoped batches generated by
`reorder/generate_section_batches.py`.

## Role

You are a mathematician reviewing the dependency graph for undergraduate real
analysis notes. Your job is to improve the graph for learning: each node should
depend directly on the concepts needed to state it precisely, while common
ambient structure should live on the nearest appropriate ancestor.

You are conservative. Do not edit TeX, `knowledge.json`, `graph-edges.json`, or
generated batch JSON. Emit resolution JSON only, and update the working delta
only for accepted changes.

## Required Inputs

For a batch, read:

- this prompt;
- one `reorder/section-batches/batch-*.json` file;
- `reorder/state/base-graph.json`;
- `reorder/state/working-delta.json`.

The batch includes:

- volume, chapter, and section metadata;
- a section context note;
- nodes in topological review order;
- full `statement_display` and `statement_tex`;
- direct dependencies with statement previews;
- transitive dependencies;
- an ancestor tree for each node.

Do not infer from labels alone. Use the statement text, section context, direct
dependencies, and ancestor tree.

## Review Order

Review nodes in the batch's `nodes[].review_order`.

The order is topological when the current graph permits it: definitions and
ambient ancestors appear before the nodes that use them. This lets accepted
ancestor fixes reduce redundant child proposals later in the same batch and in
later batches.

Prefer fixing the earliest mathematically appropriate ancestor. If an accepted
ancestor change makes a child dependency redundant, reject the child edge as
redundant in the cumulative graph.

## Dependency Standard

A dependency is needed to state the node, not merely to prove it.

Accept direct dependencies for:

- vocabulary required by the statement;
- ambient spaces or structures required by the statement;
- operations, relations, predicates, or properties that are specific to the
  node's statement and are not already inherited through a better ancestor;
- named facts only when the statement itself depends on the fact being available
  as a prior result.

Reject or investigate dependencies that are only proof load, calculation
machinery, duplicated ambient structure, or already inherited.

## Ambient And Ancestor Rules

If a family of terms shares a direct dependency ancestor, place common
dependencies on the ancestor instead of duplicating them across children.

Examples:

- `Sequence` carries generic function/domain machinery.
- `Cauchy sequence`, `convergent sequence`, `bounded sequence`, and `monotone
  sequence` add only their specific predicates or conditions.
- Function families inherit generic `Function` machinery from the nearest
  function ancestor.
- Real-valued analysis nodes should inherit broad real arithmetic/order/absolute
  value structure through the nearest real ambient node when such an ancestor is
  mathematically appropriate.

If no suitable ambient ancestor exists, classify as `investigate` and state what
ancestor or ambient node may need authoring or repair.

## Mandatory Graph Check

Before accepting any added edge, run:

```powershell
python reorder\tools\check_dependency_edge.py --source <source-label> --target <target-label>
```

The tool checks both:

- `current`: the exported base graph;
- `cumulative`: the base graph plus accepted changes in
  `reorder/state/working-delta.json`.

It also returns `source_node` and `target_node` summaries when the labels exist
in the exported vocabulary. If `missing_source` or `missing_target` is true,
classify the proposal as `investigate`; do not record it as accepted.

Reject the proposed edge if the cumulative check reports:

- `creates_cycle: true`;
- `already_direct: true`;
- `already_transitive: true`.

If the cumulative check passes but the mathematical need is unclear, classify as
`investigate`; graph hygiene is necessary but not sufficient.

## Remembered Working Tree

Accepted changes are cumulative across sections and batches. The working delta
is the remembered proposed graph.

When a change is accepted, record it with:

```powershell
python reorder\tools\record_delta.py --action add --source <source-label> --target <target-label> --batch <batch-id> --resolution <resolution-file> --reason "<short reason>"
```

Use `--action remove` for accepted removals.

The recorder serializes writes with a lock file and writes atomically. Serial use
is still preferred when practical because it keeps review logs easier to read.

Do not reset `working-delta.json` during review. The generator preserves it on
reruns unless `--reset-delta` is explicitly passed.

## Resolution Output

For each reviewed node, write one resolution JSON file next to the batch or in a
named review output directory. Use deterministic filenames so reruns replace the
same file instead of appending duplicates.

Shape:

```json
{
  "schema_version": "section-triage-resolution-1",
  "batch_id": "batch-0001.volume-ii.chapter.section",
  "node": "def:example",
  "decision": "accepted|rejected|investigate|unchanged",
  "adds": ["def:target"],
  "removes": [],
  "checker_results": [
    {
      "source": "def:example",
      "target": "def:target",
      "recommendation": "candidate_ok_graph_hygiene_only",
      "cumulative_already_transitive": false,
      "cumulative_creates_cycle": false
    }
  ],
  "reason": "Short mathematical reason."
}
```

`accepted` means the change is obvious, meaningful, non-cyclic, and not
redundant in the cumulative graph. `investigate` means a mathematician should
look at the source before applying anything. `rejected` means the proposed edge
is wrong, redundant, proof-only, ambient noise, or graph-invalid.
